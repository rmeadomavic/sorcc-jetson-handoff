QSPI bootloader update

Brand-new Jetson Orin Nano dev kits ship with old QSPI firmware that physically cannot boot a JetPack 6.x SD card. You boot a JetPack 5.1.3 SD card first, run an apt package that flashes new QSPI firmware during reboot, then swap to your JP 6.x card. Read this whole page before starting. The two traps below are the things that fail most often.

Read these first · two known traps

Trap 01 Lowercase L, not number 1.

The package name is nvidia-l4t-jetson-orin-nano-qspi-updater. That is a lowercase L, not a number 1. If you mistype it, apt says "Unable to locate package" and you will spend 20 minutes wondering why. Use tab completion: type nvidia-l4t- and press Tab.

Trap 02 Single-shot install per SD card.

The QSPI updater only fires its post-install hook the first time it is installed on a given SD card. If you reuse a JP 5.1.3 card on a second Jetson, you must add --reinstall or it will be a silent no-op and the QSPI on the new board stays untouched.

What you need

  • The Jetson, monitor, keyboard, USB-C power supply (see Section 00)
  • A microSD card (the temporary one, 32 GB+) flashed with JetPack 5.1.3
  • Internet for the Jetson (Wi-Fi or Ethernet)

Get the JetPack 5.1.3 image

  1. Go to developer.nvidia.com/embedded/downloads.
  2. Search for "JetPack 5.1.3 SD Card Image". There is an Orin Nano Developer Kit version. Download it.
  3. The file ends in .img.xz. Etcher can flash .img.xz directly. No need to extract.
  4. Open Etcher, Flash from file, pick the .img.xz, select your microSD card, Flash. Takes 10 to 15 minutes.

The procedure

01

Boot the JP 5.1.3 card.

  1. Insert the JP 5.1.3 microSD into the Jetson. The slot is on the underside of the carrier board.
  2. Plug in monitor, keyboard, USB-C power. The board boots.
  3. Walk through the Ubuntu first-boot wizard. Use sorcc for username and sorcc for password. This matches what Step 02 will use later, which keeps your muscle memory simple.
  4. Wait for the desktop to load.
02

Install the QSPI updater.

Open a terminal with Ctrl+Alt+T.

Shell · sorcc@jetson 
$ sudo apt-get update
$ sudo apt-get install nvidia-l4t-jetson-orin-nano-qspi-updater
# Triple-check the package name. l4t is lowercase L + 4 + lowercase T.

If apt says "Unable to locate package":

  • Did you run sudo apt-get update first? Required.
  • Are you on Wi-Fi that has a captive portal (hotel, coffee shop)? Switch to a hotspot or home network.
  • Is your package name actually correct? Type apt search qspi-updater to verify.
03

Reboot.

Shell · sorcc@jetson 
$ sudo reboot

The Jetson reboots, then starts flashing the QSPI firmware. The screen goes blank, or you see a UEFI prompt, and the board halts. This is exactly what is supposed to happen. Do not touch anything.

Wait at least 60 seconds after the screen goes blank.

04

Power down.

Unplug the USB-C power supply. The board is now done with the QSPI update. You can remove the JP 5.1.3 SD card and continue to Step 02.

Verifying the update worked

You cannot easily verify QSPI from JP 5.1.3. The whole point is that the version on the board does not match what is on the card. The real test: does the Jetson boot the JP 6.x card after the swap? If yes, the QSPI update worked. If no, see troubleshooting.

After you boot JetPack 6.x successfully:

Shell · sorcc@jetson 
$ cat /etc/nv_tegra_release
# Expected first line: # R36 (release), REVISION: 4.x, ...

If you see R36, the QSPI is current.

Local troubleshooting

Board does NOT halt, reboots back into JP 5.1.3 desktop

The post-install hook did not fire. Re-run with --reinstall:

Shell · sorcc@jetson 
$ sudo apt-get install --reinstall nvidia-l4t-jetson-orin-nano-qspi-updater
$ sudo reboot

Board halts, but the JP 6.x card still will not boot afterward

This is the case that bit one student in CLS 8. QSPI looked successful but JP 6 still failed. Try in order:

  1. Re-flash the JP 6.x SD card with Etcher. Use verify enabled. A bad flash is more common than a bad QSPI update.
  2. Try a different SD card. UHS-I or faster, 64 GB+. Cheap or counterfeit cards fail in weird ways.
  3. Re-do the QSPI step with --reinstall even though the first run looked successful. Costs 10 minutes, sometimes fixes it.
  4. If you have done all three: file an issue with the flash-broke.md template. Include the output of dmesg from a JP 5.1.3 boot.

The board boots straight into JP 6.x without halting

Lucky you. This Jetson already had a recent QSPI. Skip ahead to Step 02.
